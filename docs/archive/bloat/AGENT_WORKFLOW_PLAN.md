# Подробный план оптимизации и улучшения ИИ-Агента (Agent Mode)

Этот документ описывает детальный жизненный цикл работы автономного ИИ-агента (на базе философии **Arena.ai Agent Mode**), потенциальные зоны улучшения для каждого этапа и конкретные задачи по их оптимизации.

Мы можем внедрять эти доработки как последовательно, так и пакетно («за один проход»), объединяя связанные задачи для максимальной скорости и стабильности рантайма.

---

## Этап 1. Исследование и Аудит (Discovery)

### 1.1 Как устроен процесс сейчас:
- Агент получает задачу и запускает сканирование структуры проекта.
- Читаются файлы верхнего уровня (`package.json`, `README.md`, `.env.example`, `docker-compose.yml`).
- Осуществляется поиск по ключевым словам (`grep`) для понимания импортов и архитектурных связей.
- Читаются специфичные правила проекта (если они заданы в `.cursorrules` или `rules.md`).

### 1.2 Зоны для улучшения (Продуктовые задачи):
- **[A] Автоматический маппинг репозитория (Repo Map v2):** Генерация облегченного графа зависимостей и дерева импортов при старте сложной задачи, чтобы агент сразу понимал, какой файл куда импортируется.
- **[B] Глубокий аудит БД и Портов:** Перед началом кодинга агент должен проверить, какие порты заняты на хосте, живы ли СУБД (Postgres, SQLite) и соответствуют ли схемы таблиц ожиданиям (проверка миграций).
- **[C] Детектор конфликтов версий:** Автоматический анализ версий Node.js/Python в окружении перед запуском кода, чтобы избежать синтаксического несовпадения (например, ESM vs CommonJS).

---

## Этап 2. Пошаговое планирование (Planning)

### 2.1 Как устроен процесс сейчас:
- На основе собранного контекста агент разбивает задачу на логические подзадачи.
- Формируется текстовый план во внутреннем состоянии.
- План выводится пользователю на экран.

### 2.2 Зоны для улучшения:
- **[A] Строгая верификация планов (Plan Validator):** Добавление автоматического логического цензора, который проверяет:
  - Реалистичны ли шаги?
  - Не забыл ли агент шаг верификации (тестов) после шага изменения кода?
  - Учтены ли ограничения безопасности (например, запрет на прямую работу с `root` без бэкапа)?
- **[B] Динамическая перепланировка (Dynamic Replanning):** Если в процессе выполнения шага 3 агент обнаружил, что структура кода отличается от первоначальной, он должен автоматически перестроить оставшиеся шаги плана и обновить статус в UI.

---

## Этап 3. Выполнение изменений (Execution)

### 3.1 Как устроен процесс сейчас:
- Агент применяет точечные правки в файлы через `edit_file` (fuzzy-matching) или переписывает файлы целиком через `write_file`.
- Выполняет консольные команды на хосте через `host_bash` (сборка, миграции, запуск сервисов).
- Благодаря нашему недавнему обновлению, любые изменения файлов через `host_bash` теперь автоматически отслеживаются трекером изменений (`detectWorkspaceChangesAround`).

### 3.2 Зоны для улучшения:
- **[A] Автоматическое ветвление Git (Task-Scoped Branches):** Перед началом выполнения изменений агент автоматически создаёт изолированную Git-ветку (например, `task/feature-XXX`). В случае критического сбоя или если пользователь нажмет "Отмена", агент выполнит мгновенный безопасный `git reset --hard` или вернётся на стабильную ветку.
- **[B] Интеллектуальный Fuzzy-Patching:** Оптимизация алгоритма замены блоков кода в больших файлах. Если файл весит > 100 КБ, агент должен уметь применять патчи (`git apply` / `patch`) вместо перезаписи всего файла, что экономит трафик и предотвращает случайное стирание соседнего кода.

---

## Этап 4. Верификация результатов (Verification Gate)

### 4.1 Как устроен процесс сейчас:
- Система проверяет, были ли изменены файлы кода.
- Если код менялся, а агент пытается сдать задачу без запуска тестов, срабатывает блокировка (Pushback), требующая запустить верификатор (`verify_code`, `npm_test` и т.д.).

### 4.2 Зоны для улучшения:
- **[A] Автоматический линтинг и компиляция (Auto-Linter Gate):** После изменения любого JS/TS/Python файла агент автоматически, без напоминания со стороны системы, должен прогнать его через быстрый статический анализатор (например, `eslint --fix` или `tsc --noEmit`), чтобы гарантировать отсутствие глупых опечаток и ошибок типов.
- **[B] Живой Health-Check деплоя:** Если задача включала в себя деплой или перезапуск сервера, верификатор должен не просто проверить код, а сделать реальный локальный HTTP-запрос к запущенному сервису (`curl http://localhost:PORT`) и убедиться, что сервер отвечает кодом `200 OK`, а не лежит в OOM или бесконечном рестарте.

---

## Этап 5. Самопроверка и Код-Ревью (Reflection)

### 5.1 Как устроен процесс сейчас:
- Выполняется фоновый запрос `runReflectionCheck` с температурой `0.1` к ИИ-цензору.
- Цензор проверяет соответствие черновика ответа и выполненной работы изначальному промпту пользователя.

### 5.2 Зоны для улучшения:
- **[A] Консенсус двух моделей (Dual-Model Review):** Использование независимой модели для ревью. Например, если код писал быстрый локальный `Qwen-Coder`, ревьювером выступает мощный глубокий `Gemini 2.5` или `DeepSeek-R1`. Это даёт объективный взгляд со стороны.
- **[B] Аудит безопасности (Security & Leak Review):** Проверка изменённых файлов на предмет случайной утечки паролей, API-ключей, тестовых токенов или отладочных логов (`console.log`, оставленные пароли в тестовых фикстурах) перед финальной сдачей.

---

## Этап 6. Финальный отчёт и Сдача (Finalization & Report)

### 6.1 Как устроен процесс сейчас:
- Агент формирует понятный структурированный отчёт на русском языке.
- Описываются изменённые файлы и результаты тестирования.

### 6.2 Зоны для улучшения:
- **[A] Генерация артефакта сборки (Run Artifact v2):** Автоматическая запись детального JSON-файла отчёта (включая диффы всех правок, логи тестов, время работы и затраченные токены) в архивную папку `.browserai/runs/<run_id>.json` для истории и аудита.
- **[B] Авто-генерация Pull Request:** Если изменения прошли все проверки (этап 4) и ревью (этап 5), агент должен автоматически оформить красивый Pull Request в вашем GitHub-репозитории с описанием изменений и прикреплённым отчётом о тестировании.

# Наш план действий по улучшению:

Мы можем брать этот план по блокам и улучшать систему. 

**Удобство «одного прохода»:** 
Да, абсолютно! Если мы выберем конкретный сценарий (например, **«Сделать процесс изменения кода абсолютно надёжным»**), мы сможем за один проход связать задачи из разных этапов:
1. **Из этапа 3:** Авто-создание Git-ветки перед началом изменений.
2. **Из этапа 4:** Автозапуск линтера после записи файла.
3. **Из этапа 6:** Автоматическое создание Pull Request на GitHub после успешных тестов.

Агент сделает это одной красивой, монолитной архитектурной транзакцией.

С какого этапа или конкретной зоны улучшения вы хотите начать? Мы можем прямо сейчас детально проработать и внедрить выбранную вами фичу на сервер Timeweb!
